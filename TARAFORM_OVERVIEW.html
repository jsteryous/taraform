01

What Taraform is

overview

Taraform is a CRM built specifically for land acquisition — the business of buying rural / vacant parcels from their owners. It's multi-tenant out of the box, with presets for land, restaurant, and generic marketing, so the same engine can run a land-flipping pipeline and a restaurant CRM side-by-side.

The front end talks straight to Supabase for reads / writes; a thin backend server wraps the operations that need secrets — SMS sending, OAuth callbacks for Gmail / Outlook, settings PUTs. Postgres row-level security enforces tenant isolation at the database layer.

02

Property identity, not contact identity

core idea
★ The thesis
Almost every CRM keys on the person. Taraform refuses to. The stable identity of a record is the property — its APN (Tax Map ID) within a given county. People change. Owners die, sell, divorce, move, refuse to pick up, hand the parcel to an LLC. The parcel doesn't. county + tax_map_id uniquely identifies a piece of dirt forever, and Taraform treats that pair as the source of truth for "have we already talked to this thing?"

The problem this solves, visualized

Skip-trace returns 3 names for one parcel — Taraform collapses them by county + APN.

Why this matters specifically in land

↻

Lists get bought and re-bought

The same parcel will appear under a trust, then a person, then an LLC. Name-keyed dedupe would create three rows for one parcel.

☎

Phone numbers are noisy

Skip-trace returns ~5 phone candidates per row. Keying on phone would either drop leads or merge unrelated households.

⌖

APNs are scoped by county

Different counties reuse the same numbering schemes. The real key isn't tax_map_id — it's county|tax_map_id.

⤓

Re-imports enrich, not fragment

Next month's list merges new phones onto the existing parcel record — history, notes, offers, status all stay attached.

// src/lib/CLAUDE.md — explicit rule in the codebase
Tax map IDs are unique within a county, not globally.
Same parcel ID can exist in different counties — duplicate
detection must key by `county|taxMapId`, not `taxMapId` alone.

How property-keyed dedupe works

code

Two entry points apply the same rule: the single-contact Add form, and bulk CSV Import.

The import pipeline

ImportModal.jsx — three-way bucketing before any insert.

The actual code

function buildLookupMaps(contacts) {
  const byName    = new Map();
  const byAddress = new Map();
  const byTaxId   = new Map();
  for (const c of contacts) {
    if (c.firstName && c.lastName)
      byName.set(`${c.firstName.toLowerCase()}|${c.lastName.toLowerCase()}`, c);
    for (const a of c.propertyAddresses || [])
      byAddress.set(a.toLowerCase(), c);
    // Tax map IDs are only unique within a county — key by county|id so same
    // parcel ID in a different county isn't flagged as a duplicate.
    const cKey = (c.county || '').toLowerCase();
    for (const t of c.taxMapIds || [])
      byTaxId.set(`${cKey}|${t.toLowerCase()}`, c);
  }
  return { byName, byAddress, byTaxId };
}

When a parcel matches, the import doesn't skip outright — it computes the new phones and bulk-updates the existing record: 

const newPhones = c.phones.filter(p => !(existing.phones || []).includes(p));

Search uses the same identity

The list search bar ORs partial matches across name / county / owner_address / email with exact matches against jsonb arrays tax_map_ids, property_addresses, and phones — via PostgREST's JSON containment syntax cs.["value"], backed by GIN indexes. Paste an APN into the search box → snap straight to the parcel.

Subtle correctness rule: Empty jsonb arrays are [], not {}. The phone "missing" filter compares to []; getting that wrong silently returns the wrong contacts (commit 3dd4f8d).

Stack & architecture

diagram
SPA writes to Supabase directly; the backend server handles anything that needs a secret.
  • Deploy. CI on push to main → GitHub Pages. No local gh-pages step.
  • Routing. HashRouter (GitHub Pages limit). Contact detail is a full-screen overlay synced to /#/contact/:id so the back button closes it.
  • Split context. useAppData() for contacts/clients/filters; useAppUI() for toast/theme. A toast pop doesn't re-render the contact list.
  • Pagination. 50/page server-paged; loadMoreContacts appends. All filters go to Supabase as query params, never filtered in JS.
  • Cross-tab freshness. Tab focus auto-refreshes page 1 (throttled 2s) — writes from the LandID extension or another tab show up without a manual reload.

phones, tax_map_ids, property_addresses, activity_log, custom_fields All jsonb, not text[]. Containment uses cs.["value"]; the text-array form cs.{value} errors out. Backed by GIN indexes.

custom_field_definitions A TEXT column holding stringified JSON — parse via parseCustomFieldDefs, not bare JSON.parse.

email_tokens OAuth per client, provider gmail | outlook. Popup flow posts auth-success / auth-error messages back to the SPA.
LandID Chrome extension A companion extension shares the same Supabase session, so a user signed into the CRM is already authenticated for parcel-lookup tools.

CSV import auto-maps columns. county|tax_map_id check buckets each row into add, merge phones, or skip. Bulk insert in chunks of 500.

2

Outreach

SMS / Email tab on the contact. last_sms_at drives the server-side activity filters (sms_7, sms_30, sms_never).

3

Offer cycle

OffersTab: Pending → Countered → Accepted / Rejected. Contact status auto-derives from the latest offer — batch the offers + status into one onChangeMultiple() or the second write clobbers the first.

4

Close

Status manually flips to Closed or Dead/Pass. Notes & offers stay attached to the parcel record forever.

Patterns worth knowing

conventions
✎

Blur-to-save with optimistic revert

Fields update a draft on onChange and persist on onBlur via useDraftSave. On save failure the local state reverts and a toast fires — no Save button, no phantom edits on 4xx.

⚙

Client config drives the UI

resolveConfig(currentClient) returns statuses, colors, tabs, visible fields, stats pills, terminology. Never hardcode — presets exist for land, restaurant, generic.

✓

CSS / JSX sync gate

npm run check-css flags missing CSS classes and raw rem font sizes (use the --text-xs … --text-xl tokens). Both exit 1 — CI catches them.

!

useEffect TDZ landmine

Effects in AppProvider referencing loadContacts / saveContact / etc must be placed after their useCallback. Dep arrays evaluate at render time → otherwise minifies to Cannot access 'le' before initialization.

TL;DR

one paragraph
★ THE WHOLE THING
Taraform is a React + Supabase SMS/email CRM for land acquisition. The thing that separates it from a generic CRM is that it identifies records by parcel — county + Tax Map ID — not by contact name. That single decision is what lets the same parcel survive owner changes, re-imports, and skip-trace noise without fragmenting into duplicate records, and it's enforced at every entry point: manual add, bulk CSV import, and search. Everything else (jsonb arrays with GIN containment, blur-to-save with optimistic revert, split data/UI contexts, server-paged filtering, client-driven config) is plumbing in service of that identity choice.
